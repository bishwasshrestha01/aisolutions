AI Solutions — Deployment Guide

Complete guide covering hardware and software requirements, installation from ZIP file to local development with VS Code, and production deployment to PythonAnywhere.

1. Technical Requirements

This section describes the hardware specifications and software stack needed to run the AI Solutions application in both development and production environments.

1.1 Hardware Requirements

ComponentMinimum (Development)Recommended (Production)Description
Processor (CPU) Dual-core, 1.8 GHz Quad-core, 2.5 GHz or higher The application is lightweight and does not require intensive computation. A dual-core processor is sufficient for running the Django development server locally. For production, a quad-core processor ensures smooth handling of multiple concurrent HTTP requests, database queries, and AI chatbot API calls.
Memory (RAM) 4 GB 8 GB or higher Django + SQLite runs comfortably in 4 GB RAM during development. In production, 8 GB is recommended to accommodate the web server (Gunicorn), Django processes, SQLite, and the Python runtime without memory contention. PythonAnywhere's free tier provides 512 MB which is sufficient for low-traffic sites.
Storage (Disk) 500 MB free space 2 GB or higher The source code is approximately 5–10 MB. The SQLite database grows with registrations, inquiries, and demo requests. Python packages and the virtual environment consume approximately 200–400 MB. Allocate at least 500 MB for development and 2 GB for production to accommodate logs, static files, and database growth.
Network Broadband internet (1 Mbps) Broadband internet (5 Mbps or higher) Required for downloading Python packages via pip, pushing code to GitHub, cloning on PythonAnywhere, and the AI chatbot's OpenRouter API calls. A stable connection of at least 1 Mbps is sufficient for development; 5 Mbps or higher ensures fast deployment and responsive API calls in production.
Operating System Windows 10/11, macOS 12+, or Linux Linux (Ubuntu 22.04+ or Debian 11+) The application is developed on Windows and deployed on PythonAnywhere's Linux environment. Any modern OS can run the Django development server. Production is strongly recommended on Linux for stability, security, and compatibility with Gunicorn and WSGI servers.

2. Installation from ZIP File (Local Development with VS Code)

Follow these steps to set up the AI Solutions project on your local machine starting from a ZIP file. Each step includes a description of what it does and why it is necessary.

1

Extract the ZIP File Using File Manager

Description: The project is distributed as a ZIP archive containing all source files, configurations, and documentation. Use your operating system's built-in file manager to extract the contents to a location on your computer. This creates the project root directory that will serve as your workspace.

# Windows File Explorer:
# 1. Navigate to the downloaded ZIP file
# 2. Right-click the ZIP file
# 3. Select "Extract All..."
# 4. Choose a destination folder (e.g., C:\Users\YourName\Projects\)
# 5. Click "Extract"

# After extraction, the folder structure should look like:
C:\Users\YourName\Projects\aisolutions\
    manage.py

    config\
    core\

    css\
    js\
    photos\

    *.html (index.html, about.html, contact.html, etc.)
Tip: Choose a short path with no spaces (e.g., C:\Projects\aisolutions) to avoid command-line issues.

# 9. Select "Paste" (or press Ctrl+V)

5

Create a Virtual Environment

Description: A virtual environment is an isolated Python environment that keeps the project's dependencies separate from your system Python. This prevents version conflicts between different projects. The venv module (included with Python) creates a new virtual environment in a venv/ folder inside the project directory. This folder is listed in .gitignore and will never be committed to Git.

# In VS Code terminal (PowerShell), at the project root:
py -3.13 -m venv venv

# This creates the venv/ folder with:
#   venv\Scripts\python.exe     (Python interpreter)
#   venv\Scripts\pip.exe        (Package manager)
#   venv\Scripts\activate.ps1   (Activation script for PowerShell)
#   venv\Scripts\activate.bat   (Activation script for CMD)

# Activate the virtual environment (PowerShell):
venv\Scripts\activate

# Your prompt should change to show (venv):
(venv) PS C:\Users\YourName\Projects\aisolutions>

# Verify the Python path points to the venv:
where python
# Output should show the venv path first.
Critical: Always activate the virtual environment before running any Python or Django commands. If you see packages installed but Python cannot find them, you likely forgot to activate the venv. The terminal prompt should show (venv) at the beginning.
6

Install Project Dependencies

Description: The requirements.txt file lists every Python package the project needs to run. The pip install command reads this file and downloads each package and its dependencies from the Python Package Index (PyPI). All packages are installed into the active virtual environment, keeping your system Python clean.

# Make sure the virtual environment is activated (prompt shows (venv))

# Upgrade pip to the latest version:
python -m pip install --upgrade pip

# Install all project dependencies:
pip install -r requirements.txt

# Verify installed packages:
pip list
# You should see:
#   Django            5.1.3
#   djangorestframework  3.15.x
#   django-cors-headers  4.3.x
#   requests          2.31.x
#   gunicorn          22.0.x
#   (plus sub-dependencies like sqlparse, asgiref, etc.)
Troubleshooting: If you get a "pip is not recognized" error, ensure the virtual environment is activated. If a package fails to install, try installing it individually: pip install django==5.1.3.
7

Configure Environment Variables

Description: The application reads configuration settings (secret key, database credentials, API keys) from environment variables. For local development, create a .env file in the project root directory. This file is listed in .gitignore and will NEVER be committed to Git, keeping your secrets safe. The .env file is read by the shell when you run the Django server.

# In VS Code, create a new file named .env in the project root
# File → New File (Ctrl+N)
# Save as .env (Ctrl+S)

# Paste the following template and fill in your values:
SECRET_KEY=django-insecure-<generate-a-unique-key-here>
DEBUG=True
ALLOWED_HOSTS=*
OPENROUTER_API_KEY=sk-or-v1-<your-openrouter-key>
ADMIN_USERNAME=admin
ADMIN_PASSWORD=<your-admin-password>
VariableDevelopment ValueDescription
SECRET_KEYGenerate a random stringUsed by Django for cryptographic signing (sessions, CSRF tokens, password reset tokens). Keep this secret. Generate a strong key using Python: python -c "import secrets; print(secrets.token_urlsafe(50))"
DEBUGTrueWhen True, Django shows detailed error pages with stack traces, making debugging easier. Set to False in production to avoid exposing sensitive information.
ALLOWED_HOSTS*List of domain names allowed to serve the site. * allows all hosts in development. In production, set to your actual domain (e.g., bishwasshrestha01.pythonanywhere.com).
OPENROUTER_API_KEYYour API keyAPI key for the OpenRouter AI service. Required for the AI chatbot to work. Obtain a free key at openrouter.ai/keys. Without it, the chatbot uses basic keyword responses.
ADMIN_USERNAMEadminUsername for the custom admin panel login at /admin/. This is NOT Django's built-in admin — it uses session-based authentication against these environment variables.
ADMIN_PASSWORDYour passwordPassword for the custom admin panel login. Must be at least 4 characters. Change this before deploying to production.
Security: Never share your .env file, commit it to Git, or paste its contents in public. The .gitignore file already includes .env to prevent accidental commits.
8

Apply Database Migrations

Description: Django models define the database structure (tables, columns, relationships). Migrations are Python files that translate these models into actual database tables. Running python manage.py migrate applies all pending migrations to the SQLite database, creating the db.sqlite file with tables for Events, Users, Inquiries, Demo Requests, Feedback, and Page Visits.

# Ensure virtual environment is activated
# Run migrations to create database tables:
python manage.py migrate

# Expected output (similar to):
#   Operations to perform:
#     Apply all migrations: admin, auth, contenttypes, core, sessions
#   Running migrations:
#     Applying core.0001_initial... OK
#     ...

# This creates db.sqlite in the project root:
# - Events table: stores event details (title, date, capacity, etc.)
# - Users table: stores user information (name, email)
# - Inquiries table: stores contact form submissions
# - DemoRequests table: stores demo scheduling requests
# - Feedback table: stores user feedback and ratings
# - PageVisits table: stores daily visit counts
# - EventRegistrations table: stores event registration records
Note: Run python manage.py migrate every time you pull code changes that include new or modified models. If you see "No migrations to apply," the database is already up-to-date.

Running this in development ensures the configuration is correct before deployment.

# Create a repository on GitHub:
# 1. Go to https://github.com/yourusername
# 2. Click the "+" icon → "New repository"
# 3. Name it "aisolutions"
# 4. Do NOT initialize with README (we already have files)

import os
import sys

# Project path

if project_home not in sys.path:
    sys.path.insert(0, project_home)

# Activate virtual environment automatically
activate_this = project_home + '/venv/bin/activate_this.py'
if os.path.exists(activate_this):
    with open(activate_this) as f:
        exec(f.read(), {'__file__': activate_this})

os.environ['DEBUG'] = 'False'

6. Application Architecture Overview

Understanding how the application is structured helps with debugging and making changes.

6.1 Architecture Flow

# Request Flow:
Browser (user)  →  PythonAnywhere Web Server  →  WSGI File  →  Django
  →  URL Router (config/urls.py)  →  Django View / API View
  →  Database (SQLite) or HTML File  →  Response back to browser

# Frontend (HTML + JavaScript):
- Static HTML pages (index.html, about.html, contact.html, etc.)
- JavaScript uses Fetch API to call Django REST endpoints
- Bootstrap 5 for responsive design
- Chatbot widget embedded in pages

# Backend (Django + DRF):
- REST API at /api/* for all data operations
- Custom session-based admin authentication at /admin/
- AI Chatbot via OpenRouter API at /api/chat/
- Visit tracking via middleware

6.2 Key Files Reference

FilePurpose
config/settings.pyAll Django settings: databases, apps, middleware, CORS, REST framework, session config, admin credentials, and OpenRouter API configuration.
config/urls.pyRoot URL configuration: routes API requests to core/urls.py, serves static HTML pages, and handles the admin panel routing.
config/wsgi.pyWSGI entry point for production servers. Used as the base for PythonAnywhere's WSGI file.

core/models.pyDatabase models: Event, EventRegistration, Inquiry, User, PageVisit, DemoRequest, Feedback.
core/urls.pyAPI endpoint routing: maps URLs to view functions for all 20+ endpoints.

chatbot/knowledge.pyAI chatbot logic: system prompts, pricing responses, keyword-based fallback responses, response sanitization.
admin/*.htmlAdmin panel frontend: 9 HTML pages for dashboard, analytics, events, registrations, inquiries, feedback, demos, and settings.
